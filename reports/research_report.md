## 3. 核心问题：物资名称 & 需求数量分析

### 3.1 列表API字段清单

**列表API已有的字段**（可直接获取）：

| 字段 | 英文名 | 示例值 |
|------|--------|--------|
| 公告标题 | `title` | 国网冀北电力有限公司2026年第四次**物资**公开招标采购 |
| 项目编号 | `code` | JB26-HW-ZB04-0126W4 |
| 发布单位 | `publishOrgName` | 国网冀北电力有限公司 |
| 发布时间 | `noticePublishTime` | 2026-06-26 |
| 公告类型 | `noticeType` | 100063001 (公开招标) |
| 文档ID | `firstPageDocId` | 2606268991036031 |
| 公告URL | (可构造) | `/ecp2.0/portal/#/doc/doci-bid/...` |

**列表API缺失的关键字段**：

| 缺失字段 | 重要性 | 所在位置 |
|----------|--------|----------|
| ❌ **物资名称** | 核心 | 详情页/货物清单附件 |
| ❌ **物资描述/规格** | 核心 | 货物清单附件(Excel) |
| ❌ **需求数量** | 核心 | 货物清单附件(Excel) |
| ❌ **计量单位** | 重要 | 货物清单附件 |
| ❌ **包号** | 重要 | 详情页 |
| ❌ **限价/预算** | 重要 | 详情页 |
| ❌ **中标企业** | 核心 | 中标公告 |
| ❌ **中标金额** | 核心 | 中标公告 |

### 3.2 物资名称的获取路径

```
招标公告列表 (noteList API)       ← 无物资信息
        │
        ▼
招标公告详情页 (SPA渲染)          ← 含物资大类名称
        │
        ▼
货物清单附件 (Excel/PDF/ZIP)      ← 含逐项物资名称+数量
```

**路径A**：从公告标题提取物资大类
- 标题如"国网冀北电力有限公司2026年第四次**物资**公开招标采购"
- 可区分：物资类 / 工程服务类 / 零星服务类 / 框架类
- ❌ 无法区分具体物资名称（如"10kV变压器"、"组合电器GIS"）

**路径B**：解析详情页HTML
- 需使用Selenium/Playwright渲染Angular SPA
- 详情页包含分标名称、包号等结构化表格
- ⚠️ 401条 × 每条渲染 ~3-5秒 = 20-35分钟
- ⚠️ 需处理SPA异步加载、反爬机制

**路径C**：下载并解析货物清单附件
- 公告底部通常有".zip"或".xlsx"附件
- 附件含逐项：物料编码、物资描述、单位、数量、限价
- ✅ 数据最完整
- ⚠️ 需模拟浏览器下载，解析Excel

### 3.3 需求数量的获取策略

| 数据层级 | 数量信息 | 粒度 | 获取难度 |
|----------|---------|------|---------|
| 公告标题 | 无 | - | - |
| 详情页 | 分包汇总数 | 包级别 | 中 |
| 货物清单 | 逐项需求数量 | 物料级别 | 高 |
| 中标公告 | 中标数量 | 包级别 | 中-高 |

## 4. 技术可行性评估

### 4.1 第一阶段：列表数据采集（可行 ✅）

```
难度：★☆☆☆☆（低）
数据量：401条（冀北）/ 14,329条（全平台）
时间：~30秒（全量列表采集）
字段：10个基本字段
限制：无物资名称、无数量
```

### 4.2 第二阶段：标题分类筛选（可行 ✅）

```python
# 从标题中可提取的维度：
- 物资/服务/工程 大类分类（关键字匹配）
- 招标批次（如"第四次"、"第五批"）
- 项目年份（如"2026"）
- 采购方式（公开招标、单一来源等）

# 冀北401条公告中：
物资类关键词匹配: ~16/60 (27%)
服务/工程类匹配:   ~24/60 (40%)
```

### 4.3 第三阶段：详情页采集（可行但有挑战 ⚠️）

```
难度：★★★☆☆（中）
方式1：Selenium/Playwright 模拟浏览器
  - 优点：完整渲染SPA，获取所有DOM内容
  - 缺点：慢（3-5秒/条），占用内存
  - 时间：401条 ≈ 20-35分钟

方式2：分析 SPA 内部 API 调用
  - 优点：直接获取JSON数据（如找到详情API）
  - 缺点：接口可能加密/需token
  - 时间：取决于逆向分析结果

方式3：混合策略
  - 先用列表API获取全量元数据
  - 仅对"物资类"公告（~110条）做详情采集
  - 减少 73% 的工作量
```

### 4.4 附件下载解析（可行但需投入 ⚠️）

```
难度：★★★★☆（中-高）
- Selenium点击下载附件 → 等待ZIP/Excel文件
- 解析Excel：提取物料编码、名称、数量、单位、限价
- 数据质量高，但流程复杂
```

## 5. ECP平台数据 vs EPTC报告对比

| 维度 | ECP原始数据 | EPTC付费报告 |
|------|-----------|-------------|
| 物资名称 | 需从附件提取 | ✅ 已结构化分类（19类） |
| 需求数量 | 需从附件提取 | ✅ 已汇总统计 |
| 中标企业 | 需解析中标公告 | ✅ 含企业排名和份额 |
| 各省对比 | 原始数据分散 | ✅ 已汇总对比 |
| 五年趋势 | 需自行汇总 | ✅ 已做趋势分析 |
| 获取成本 | 免费（需爬取） | ¥2,800~3,600/册 |
| 数据粒度 | 物料级别 | 分类级别 |
| 时间跨度 | 约6年 | 5年滚动 |
| 维护成本 | 持续爬取 | 年度购买 |

**结论**：EPTC报告本质上就是对ECP原始数据的加工品。对于需要**物料级别详细数据**的场景，ECP原始数据更优；对于需要**快速获取汇总分析**的场景，EPTC报告更省力。

## 6. 最终结论

### 6.1 直接回答用户问题

| 问题 | 答案 |
|------|------|
| ECP数据能否区分**物资名称**？ | ⚠️ 列表API不能，需从详情页附件提取。标题可区分物资大类 |
| ECP数据能否获取**物资需求数量**？ | ⚠️ 列表API不能，需从货物清单附件(Excel)提取 |
| ECP数据是否满足EPTC报告级别的分析需求？ | ⚠️ 原始数据足够，但需大量加工处理 |

### 6.2 推荐方案

**短期方案（1-2天）**：
1. 使用本工程 `ecp_client.py` 爬取冀北401条公告列表
2. 基于标题关键词筛选物资类公告（~110条）
3. 手动抽样下载几个货物清单附件，验证附件数据结构
4. 使用Playwright快速渲染几个详情页，确认DOM结构

**中期方案（1-2周）**：
1. 实现Playwright详情页爬虫，提取物资大类+包号
2. 实现附件自动下载+Excel解析管道
3. 建立本地SQLite数据库存储结构化数据
4. 实现数据去重和增量更新

**长期方案（1月+）**：
1. 扩展至全平台27省公司
2. 建立中标公告匹配（物资→中标企业→中标金额）
3. 自动生成供需分析报表
4. 替代/补充EPTC付费报告
