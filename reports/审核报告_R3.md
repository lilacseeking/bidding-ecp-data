## 一、文档间一致性检查

### 1.1 核心矛盾审查："低确定性步骤 VMD 25%" vs "不优先实施残差 VMD"

**审查结论：两个文档高度一致，无实质性矛盾。**

| 维度 | 步骤确定性评估.md | 模型合理性评估.md | 一致性 |
|------|-----------------|------------------|:------:|
| VMD 当前状态 | 步骤10标记为"低确定性（25%）"，4个模型已废弃，仅保留基础函数 | 第2节确认4个VMD模型全部废弃，基础组件可复用 | 一致 |
| VMD 替代方案 | 方案A: VMD去噪 / 方案B: VMD分频预测 / 方案C: 放弃VMD | 第6.4.3节替代方案2: VMD去噪（"正确用法"）；第6.5节结论"不优先实施残差VMD" | 一致 |
| 残差VMD 实施优先级 | 作为"阻塞点1"标记为【必须解决】，推荐先验证方案A | 第6.5节标记为P2（最后实施），"不优先实施" | **表面矛盾，实质互补** |
| 实施时序 | "第2-3周: VMD方法学实验（先验证方案A: VMD去噪）" | "先完成批次事件特征+CatBoost增强（方案7）和N-HiTS（方案5），如果MASE仍>1，再考虑残差VMD" | **需要统一** |

**详细分析**：

步骤评估将 VMD 方法学重设计标记为"阻塞点1（关键路径）"——"不解决则论文不能发表"。模型评估则将残差VMD标记为 P2（最后实施）——"不优先实施"。表面看有矛盾，但实际上：

- 步骤评估的"阻塞点1"指的是**必须找到一种不构成循环论证的VMD用法来替代当前废弃的4个模型**。这确实是阻塞性的——如果VMD完全废弃且没有替代方案，论文方法论会有缺口。
- 模型评估的"不优先实施"指的是**残差VMD（用户提出的具体方案）不应该作为首要方案**，因为该范式已有文献先例且在当前数据上可行性存疑。
- 两者的共同出路是：**先验证批次事件+N-HiTS能否让论文方法论成立，如果能则不需要VMD；如果效果不足，再启用VMD去噪方案**。

**判定：无实质性矛盾，但需要在论文汇总文档中明确统一实施策略。**

## 二、推荐模型的可实现性审查

### 2.1 N-HiTS 的实现路径

**审查结论：步骤评估给出了N-HiTS的实现路径，但不够具体。**

**步骤评估中的覆盖**：
- 步骤12 "对比模型构建"中 N-HiTS 未被单独列出（对比模型列的是 SARIMA/LSTM/XGBoost/Naive Seasonal）
- 步骤11 "Transformer时序预测"中提到"将Transformer用于不依赖VMD的直接多变量时序预测"
- 步骤9 "CatBoost Baseline"中 N-HiTS 未被提及

**模型评估中的覆盖**：
- 第3.1节推荐 N-HiTS 为⭐⭐⭐⭐
- 第3.2节 "推荐二"："N-HiTS（P1，短期可实现），Nixtla的neuralforecast库直接支持，一行调用即可"
- 第3.4节实现可行性：N-HiTS 预计1-2人日，约30行新增代码

**缺失**：步骤评估的17步分解中**没有单独的N-HiTS实现步骤**。步骤12"对比模型构建"的细节中也没有包含N-HiTS。N-HiTS目前仅出现在模型评估的推荐中，尚未转化为可执行的步骤。

**建议**：在步骤12或新增一个子步骤中，补充N-HiTS的实现路径。

### 2.2 N-HiTS 需要安装什么库？Nixtla的neuralforecast是否支持外生变量？

**技术验证结果**：

1. **安装库**：`pip install neuralforecast`（Nixtla官方库，PyPI包名 `neuralforecast`）
   - N-HiTS 类在 `neuralforecast.models.NHITS`
   - 模型评估中"一行调用"的描述基本准确：`NHITS(h=12, input_size=36, ...).fit(df).predict()`

2. **外生变量支持（关键问题）**：**Nixtla的neuralforecast原生支持三类外生变量**：
   - `futr_exog_list`：未来已知的外生变量（如批次安排、日历特征、春节距天数）——这是批次事件特征的直接入口
   - `hist_exog_list`：历史观测到的外生变量（如铜铝价格、用电量）
   - `stat_exog_list`：静态外生变量（如物资类别编码）
   - Nixtla 还发布了 NBEATSx 版本（专门为外生变量扩展的 N-BEATS），N-HiTS 继承同一框架

3. **与论文的适配性**：
   - 批次事件特征（未来的确定性信息）可通过 `futr_exog_list` 直接输入
   - 铜铝价格等历史因子可通过 `hist_exog_list` 输入
   - 81个月数据对N-HiTS足够（模型评估中"低数据需求"的判断正确）

**判定**：N-HiTS 的技术可行性已确认。但步骤评估和模型评估均未明确N-HiTS与外部因子的具体接口代码，这应该在实施阶段补充。

### 2.3 批次事件特征需要从 ECP DB 提取哪些具体字段？代码改动量多大？

**审查结论：模型评估给出了特征名称但未给出具体的数据库字段映射和SQL，步骤评估给出了"纯SQL/Pandas聚合"的简化描述。**

**已知信息**（来自模型评估）：

| 特征名称 | 数据来源表 | 是否存在 | 提取难度 |
|---------|-----------|---------|---------|
| is_bid_batch_month | bid_notices | notice_publish_time 字段已有 | 低 |
| batch_type | bid_notices | batch_info 字段有批次类型信息 | 低 |
| days_since_last_batch | bid_notices | 需按物资分组计算 | 低 |
| cumulative_batches_this_year | bid_notices | 需按年汇总 | 低 |
| batch_count_by_month | bid_notices | 需按月聚合 COUNT | 低 |
| batch_gini | bid_notices | 需滚动窗口计算（逻辑需澄清） | 中 |

**缺失的关键信息**：

1. **bid_notices 与 bid_items 的关联键**：批次事件特征是针对公告还是针对物资？如果是后者，需要 bid_notices -> bid_items 的 JOIN（关联键：notice_id? notice_title?）
2. **batch_info 字段的格式和价值**：模型评估提到 `batch_info` 字段，但未说明其具体格式（JSON? 纯文本？批次编码？）。如果不能可靠解析，batch_type 特征可能无法生成。
3. **物资-批次的关联逻辑**：一种物资可能出现在多个批次中，不同批次的公告发布时间不同。`days_since_last_batch` 需要按物资分组计算——这需要确认 bid_items 表中每条记录的 `notice_id` 或等价字段是否存在。
4. **batch_gini 的逻辑**：R2 审核已经指出"'当月Gini'不可计算"，需要定义具体窗口。

**代码改动量估算**：

| 组件 | 文件/模块 | 预估改动 |
|------|----------|---------|
| SQL 查询 | 新增 batch_features.sql 或在 fetch_external_factors.py 中增加 | ~50行 SQL |
| 特征编码 | 在 preprocess_data() 中增加 batch 特征编码函数 | ~80-100行 Python |
| 总改动 | | **~150行新代码，1-2人日** |

这与步骤评估中步骤7的"1-2人日（依代码复杂度）"和模型评估中"1-2人日"的估算一致。

**缺失**：两个文档都没有给出具体的 SQL schema 和 Python 函数签名。在实施前应该核实以下字段的实际存在性：
- `bid_notices.notice_publish_time`（已确认存在）
- `bid_notices.batch_info`（存在但格式未经验证）
- `bid_items.notice_id` 或等价的外键（未确认是否存在，决定是否可以做物资维度的批次特征）

## 五、审核结论

### Phase 3 审核通过，可以进入 Phase 4 最终汇总。

Phase 3 的两个文档（步骤确定性评估 + 模型合理性评估）完成了以下核心工作：

1. **为论文提供了完整的17步实施路线图**，覆盖数据 -> 因子 -> 特征 -> 模型 -> 评估 -> 写作的全链路
2. **3个关键阻塞点均已有明确的解决路径**（VMD方法学通过方案A/B/C备选；Excel修复方案已定；外部因子有P0/P1/P2三层路线图）
3. **模型方案从"废弃什么"到"推荐什么"形成了闭合的逻辑链**（废弃4个VMD循环论证模型 -> 推荐批次事件驱动+N-HiTS -> 为残差VMD提供了严谨的可行性分析和查重报告）
4. **综合可行性评估72%（修复后85%+）的判断谨慎但乐观**，对风险的认知充分

### 进入 Phase 4 的条件

Phase 4 分两步走：

- **Phase 4a（前置）**：先完成论文大纲 + 数据统计汇总表（0.5-1人日）
- **Phase 4b（最终汇总）**：整合11篇文档的结论，生成最终执行计划和论文写作指南（1-2人日）

Phase 4 的输出建议为一个合并文档（或2个：执行计划 + 写作基准），而非多个分散文档——这是"最终汇总"区别于前面各Phase 的关键。

### 与 R1/R2 的延续性

R1 审核了 Phase 1 的 4 篇文档（6个矛盾/错误），R2 审核了 Phase 2 的 3 篇文档（6个矛盾/错误，含2个P0）。R3 审核中未发现 Phase 3 文档存在 **R1/R2 级严重的矛盾或错误**——特别是，**影响因子增强方案 vs 代码评估的"废弃"矛盾（R2中的D1/D4/P0问题）没有传导到 Phase 3 文档中**——步骤评估和模型评估都在开篇就确认了 4 个 VMD 模型已废弃。这说明 Phase 2 审核的纠正效应已生效，文档质量在迭代中提升。
