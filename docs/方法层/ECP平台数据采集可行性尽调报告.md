# ECP平台数据采集可行性尽调报告

> 调研日期：2026-07-17～2026-07-18  
> 调研对象：国家电网新一代电子商务平台（ECP 2.0）  
> 调研目的：评估从ECP平台采集国网公司及冀北公司项目预算/中标数据的可行性

---

## 一、项目背景与调研目标

### 1.1 背景

本项目旨在为国网冀北电力有限公司提供物资采购数据采集与分析能力，覆盖招标→中标→采购全链路。现有数据（`data/ecp_data.db`）已包含 40,773 条物资明细（来自已下载的招标/中标公告文档），但缺少**项目预算金额**这一关键字段。

### 1.2 调研目标

1. 验证ECP平台各类公告的API可访问性
2. 评估从平台获取项目预算/金额数据的可行性
3. 完成中标候选人公示、中标结果公告的结构化数据采集
4. 验证数据完整性与质量

### 2.3 核心API接口

#### 接口1：公告列表查询 `POST /index/noteList`

**请求参数**：
```json
{
  "index": 1,
  "size": 20,
  "firstPageMenuId": "2018060501171111",
  "orgId": 2019040100044796,
  "key": "",
  "orgName": "国家电网有限公司"
}
```

**关键细节**：
- `orgId` 必须是 JSON **数字类型**（非字符串），否则返回空结果
- 返回结构：`resultValue.noteList`（公告数组）、`resultValue.orgList`（单位列表）
- **无 totalCount 字段**（候选人公示），需要逐页遍历直到无数据
- 结果公告有 `resultValue.count` 字段

#### 接口2：公告详情获取 `POST /index/getNoticeWin`

**请求参数**：POST body 为公告ID的 **原始字符串**（非 JSON 对象）

```
POST body: "2607169366873719"
Content-Type: application/json
```

**返回结构**：`resultValue.notice.CONT`（HTML内容）、`resultValue.fileFlag`（附件标识）

## 四、项目预算可行性分析

### 4.1 招标公告中无预算信息

通过 menuId `2018032700291334`（招标公告及投标邀请书）获取的招标公告HTML内容中：
- **不含项目预算/预估金额字段**
- 仅包含投标邀请信息、投标人资质要求、业绩门槛等
- 已通过 `grep` 在全部已转换的MD文件中搜索"预算""金额""万元"等关键词，确认**零匹配**

### 4.2 候选人公示含报价但无预算

- 候选人公示包含每个候选人的**投标报价（万元）**，这是投标方的报价
- 不含项目预算/招标控制价/标底
- 投标报价可作为**项目金额的代理指标**（尤其是排名第1的中标候选人报价通常接近最终中标价）

### 4.3 结果公告不含金额

- 中标结果公告仅包含中标人名称和中标状态
- **不包含中标金额**
- 部分"成交"类公告表格结构不同，导致约49%记录的列映射存在偏差

### 4.4 预算数据可行性结论

| 数据来源 | 是否含预算 | 是否含金额 | 可用性 |
|----------|-----------|-----------|--------|
| 招标公告 | ❌ | ❌ | 不可用 |
| 中标候选人公示 | ❌ | ✅ 投标报价 | 部分可用（仅当前公示期内） |
| 中标结果公告 | ❌ | ❌ | 仅含中标人 |
| 竞价公告 | ⬜ 未调研 | ⬜ 未调研 | 待探索 |
| 采购公告 | ⬜ 未调研 | ⬜ 未调研 | 待探索 |

**结论**：ECP平台公开数据中**无法直接获取项目预算**。候选人公示中的投标报价是唯一可用的金额数据，但受限于3-5天公示期，历史数据无法追溯。

## 五、技术验证

### 5.1 API可靠性

- ✅ `noteList` 接口稳定，支持翻页遍历全量公告
- ✅ `getNoticeWin` 接口返回完整HTML（候选人公示可达24万字符，结果公告可达20万字符）
- ✅ 无需认证，公开访问
- ✅ 无明显的反爬/限流机制（测试期间未触发封禁）

### 5.2 Playwright渲染验证

使用 Playwright headless Chromium 打开ECP页面进行对比验证：

| 测试项 | 结果 |
|--------|------|
| 列表页渲染 | SPA页面在headless模式下仅显示"智能客服"浮窗 |
| 详情页渲染 | 无法渲染内容区域 |
| API拦截 | headless模式下未触发API请求 |
| **结论** | API直接返回完整HTML，无需浏览器渲染 |

**关键结论**：ECP平台虽然是Angular SPA，但API直接返回完整的HTML内容，**不存在浏览器独有的数据**。纯HTTP请求即可获取全部可用数据。

### 5.3 数据完整性验证

通过对比API返回HTML与浏览器渲染结果：
- 候选人公示：HTML含完整表格（12列，52行样本）
- 结果公告：HTML含完整表格（6列，96-114行样本）
- **确认API数据完整性与浏览器端一致**

## 六、已知数据质量问题

### 6.1 结果公告49%记录缺失中标人

**现象**：37,530条结果公告记录中，18,313条（48.8%）中标人字段为空

**根因**：不同类型的结果公告表格结构存在变体：
- 标准"中标"公告：6列（分标编号|分标名称|包号|中标状态|项目单位|中标人）
- "成交"类公告：列数或列顺序不同
- 当前列映射仅适配标准6列结构

**待修复**：需识别"成交"类公告的表格结构并添加对应列映射

### 6.2 物资品类字段污染

**现象**：
- "中标"出现在分标名称字段：约4,010条
- "包 1""包1""包 2"出现在分标名称字段：约1,221条

**根因**：部分表格列映射偏差导致数据错位

### 6.3 候选人公示历史数据缺失

**现象**：仅能采集当前公示期内的公告（3-5天窗口）

**影响**：无法获取历史候选人公示的投标报价数据

## 八、结论与建议

### 8.1 可行性结论

| 维度 | 评估 | 说明 |
|------|------|------|
| API可访问性 | ✅ 完全可行 | 公开接口，无需登录 |
| 数据完整性 | ⚠️ 部分可行 | 候选人公示历史数据不可得，结果公告约51%有完整中标人 |
| 预算数据 | ❌ 不可行 | 平台公开数据中无项目预算字段 |
| 投标报价 | ⚠️ 有限可行 | 仅当前公示期内的候选人公示含报价 |
| 中标人/物资品类 | ✅ 可行 | 结果公告含完整物资品类和中标人（标准结构下） |

### 8.2 下一步建议

1. **修复结果公告列映射**：识别"成交"类公告的表格结构变体，补全49%缺失中标人的记录
2. **补全候选人公示项目负责人字段**：表格实际有12列，第12列"项目负责人"尚未提取
3. **探索竞价公告**（menuId `2018052500910172`）：可能包含成交通知书和金额数据
4. **探索采购公告**（menuId `2018032900295987`）：评估是否包含预估金额
5. **建立定时采集机制**：候选人公示仅3-5天有效，需定期采集避免数据丢失
