# 电力物资需求量预测——多智能体协作规范

## 1. 项目背景

本项目研究**配电网电力物资月度需求量预测**，数据来源为国网新一代电子商务平台（ECP2.0）冀北/全国物资采购公告，最终产出为一篇期刊论文。

### 项目结构

```
bidding-ecp-data/          # 上游：数据采集 + 分析
├── src/pipeline.py         # 主流水线：采集→解析→入库→统计
├── src/demand_stats.py     # 需求统计 + 可视化
├── src/db/schema.py        # 数据库表结构（SQLite）
├── build_real_data.py      # 从DB生成模型训练数据
├── data/ecp_data.db        # SQLite（4万+物资明细）
├── docs/                   # 结构化文档（数据层/方法层/工程层/决策层/文献）
├── reports/                # 分析报告与审查记录
└── outputs/figures/        # 可视化产出

vmd-catboost/               # 下游：建模实验
├── main.py                 # 模型训练 + 评估主程序
└── data.xlsx               # build_real_data.py 生成的训练数据
```

### 关键已知结论

- **VMD系列全部废弃**：ΣIMF≈y 的循环论证，不能用于论文
- **冀北数据不可用**：零值率62-72%，改用国网全国聚合数据
- **外部因子（铜铝期货/工业用电/农历编码）无效**：|ρ|<0.26，已删除
- **批次事件特征为唯一有效外部因子**：transformer_bids（|ρ|=0.66）
- **推荐模型路线**：CatBoost baseline + TwoStage（Stage1分类/Stage2回归）+ 批次事件条件预测
- **必须修复5个致命问题**（F1-F5）：数据泄露、demand_month语义、单位归一化等

## 2. 智能体角色定义

当任务涉及多视角判断或需要独立审核时，应使用 `Agent` 工具启动对应角色。以下角色按需要调用，不必每次都全部启动。

### 2.1 研究员（Researcher）

**职责**：方法论分析、文献对比、因子可行性评估、统计验证

**何时启动**：
- 需要评估某个方法的理论正确性
- 需要对比文献中的方法与本项目的异同
- 需要验证某个因子/特征是否有预测力
- 需要从第一性原理出发分析问题

**提示词要点**：给出明确的判断标准（如"必须给出Spearman ρ值"、"必须检查循环论证"），要求其输出结构化的分析表格和明确结论。

### 2.2 工程师（Engineer）

**职责**：代码实现、Bug修复、数据管道维护

**何时启动**：
- 实现新模型（如 N-HiTS、TFT）
- 修复数据管道问题
- 大规模代码重构
- 独立实现某个功能模块后需要交叉验证

**提示词要点**：指定具体的文件路径、函数名、输入输出规格。要求其先读代码再改，避免引入新的数据泄露。

### 2.3 审核员（Reviewer）

**职责**：代码审查、方法论把关、论文逻辑审查

**何时启动**：
- 工程师完成实现后，需要独立审查
- 论文章节写完后，需要检查逻辑一致性
- 怀疑存在数据泄露或方法论错误时

**提示词要点**：要求其"带着找问题的目的"审查，必须输出具体问题列表（附代码行号或文档段落），不得仅说"通过"。

### 2.4 论文作者（Paper Writer）

**职责**：论文章节撰写、修改、润色

**何时启动**：使用 `academic-paper` skill 完成论文相关工作。对于独立章节起草，可启动 Agent 先起草再汇入主文档。

## 3. 工作流模式

### 3.1 标准研究循环

```
研究员（分析问题/评估方法）
  → 工程师（实现方案）
    → 审核员（独立审查）
      → 通过 → 记录结论到 docs/ + memory
      → 不通过 → 工程师修改 → 审核员再审
```

**实际使用方式**：在对话中依次扮演上述角色，或在关键决策点启动独立 Agent 进行"第二意见"审查。不需要每次都跑完整循环，简单的修改直接做即可。

### 3.2 论文工作流

```
数据准备 → 模型实验 → 指标汇总 → 论文起草 → 审核修改 → 定稿
```

使用 `academic-paper` skill 进入论文写作模式，使用 `data-to-paper` skill 运行端到端流程。

### 3.3 门禁规则

以下节点必须通过审核才能继续：
- **方法论方向变更**（如新增/删除某种模型路线）：必须经研究员分析 + 审核员确认
- **论文定稿前**：必须经审核员全量逻辑一致性审查
- **数据管道修改**：必须确认无数据泄露（train/test 隔离、scaler fit 范围、lag 构造顺序）

## 5. 文档仓库规范

### 目录结构

```
docs/
├── index.md                 # 文档导航（每次增删文档必须更新）
├── 数据层/                   # 数据选择、验证、稀疏度分析
├── 方法层/                   # 因子验证、模型评估、方法对比
├── 工程层/                   # 代码评估、步骤拆解、模型说明
├── 决策层/                   # 执行方案、创新方案、进展记录
├── 文献/                     # 已读文献笔记
└── 早期记录/                 # 历史审核报告、会议纪要（不出现在导航中）
```

### 维护规则

1. **新分析文档**放对应层级目录，文件名用中文描述性标题
2. **更新 docs/index.md**：任何文档增删改必须同步更新文档地图
3. **每篇文档头部标注**：生成日期、置信度（[确定]/[高置信]/[待验证]）
4. **早期记录**中的内容不出现在主导航，作为历史存档
5. **结论变更时**：在文档中标注"已废弃"并指向新结论所在文档，不直接删除

### reports/ 目录

存放单次分析报告和审查记录，与 docs/ 的区别：
- `docs/` = 结构化、持续维护的知识库
- `reports/` = 一次性分析报告、会议记录、审查输出

## 7. 第一性原理护栏

在以下节点必须调用 `first-principles` 技能进行检验：

- 提出新的模型方案时
- 引入新的数据源或因子时
- 修改 train/test 划分策略时
- 论文定稿前的最终审查

### 本项目的核心第一性原理问题

1. 物资需求本质由什么驱动？（检修计划、工程扩张、设备更换周期——而非铜铝期货价格）
2. 预测的最终用途是什么？谁在用这个预测？（物资部门做采购计划——可接受一定误差，但低报比高报代价更大）
3. 什么数据是可用的？（ECP公告中的"需求量"字段，而非公告发布日期）
4. 不准确的代价是什么？（库存积压 vs 供应短缺——非对称损失）
5. 模型是否在"作弊"？（用未来信息预测未来 = 数据泄露；用 y 的分解分量预测 y = 循环论证）
