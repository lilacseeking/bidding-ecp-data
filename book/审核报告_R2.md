## 一、审查：大类vs品类相关性分析.md

### 总体评价

这是一篇数据驱动含量高的分析报告。Spearman 秩相关系数方法论正确，从 `material_demand_total` 表中直接查询的数据可信度高。但存在若干需要修正的表述问题和与 Phase 1 文档的联动缺失。

### ✅ 通过的审查点

1. **Spearman 方法论选择正确**：相比于 Pearson（对异常值敏感、要求线性关系），Spearman 更适合物资需求的非正态分布特征。且区分"全月"和"非零月"两种计算方式是关键的审慎做法，揭示了"全月 rho 具有欺骗性"这一重要发现。

2. **三类物资的逐一分析有数据支撑**：rho 值、p 值、非零月数、占比均值、CV 均来自数据库查询。占比例稳定性分析的 CV 阈值判定（>0.5 不稳定，>1.0 极不稳定）合理。

3. **大类内部子品类 Spearman 分布分析**是文档中最有洞见的部分。25 个配对（避雷器）+ 66 个配对（互感器）+ 153 个配对（绝缘子）覆盖全面，中位数分别为 0.17、0.05、0.34 —— 这三个数字本身就是有力的证据。

4. **综合结论"大类需求规律不能可靠地代表具体品类需求规律"** 有充分的统计证据支持。

5. **层次化预测可行性评估表**（第六节）的 3 个场景划分（>80%/30-80%/<30%）有数据依据，边界明确。

### 🔶 需要改进的问题

1. **[遗漏：未引用 Phase 1 文档的同类结论]** 该文档的结论与《数据稀疏度与合并方案.md》高度一致——两者都得出"不建议合并"的结论，但理由不同：

   - 大类分析认为"合并因为内部子品类不相关而无意义"
   - 数据稀疏度分析认为"合并因为零值模式重叠而无改善"

   两个论证路径互为补充，但文档之间没有交叉引用。大类分析更是完全没有提及 Phase 1 的任何文档。

   **建议**：在文档的"建议"节（第六节）开头增加一段交叉引用说明：
   > "本结论与 Phase 1 数据稀疏度分析的结论一致（不推荐合并物资大类），但论证路径不同。数据稀疏度分析表明合并大类对零值率改善有限（避雷器 0pp，互感器 3pp），本分析进一步证明即使数据层面合并，大类内部子品类需求节奏的不一致性也会使合并序列失去业务解释价值。"

2. **[遗漏：未讨论需求来源结构差异]** 文档的业务逻辑解释（第五节）提到"不同品类有不同的采购周期和采购主体"，但没有提供数据支撑。例如，`bid_items` 表中有 `project_org_name` 字段——是否可以从数据库查询不同子品类的采购主体分布来佐证这一解释？

   **建议**：可选——跑一个 quick query 看三种代表性品类在 `project_org_name` 上的分布。如果分布差异显著，加入数据支撑会增强说服力。

3. **[置信度标注遗漏]** 关键结论"大类需求规律不能可靠地代表具体品类需求规律"缺少置信度标注。虽然该结论在数据层面有充分支撑，但结论本身是对"类到品"的推广性的否定判断，建议标注置信度：

   **建议**：在"综合结论"处增加一句"**[高置信]** 该结论基于 81 个月跨度下 3 个大类 37 个子品类的完整统计。"

4. **[表达问题]** 第六节"裁决"表中"交流避雷器"的全月 rho=0.97、非零月 rho=0.90，裁决为"大类规律**可以**代表（该品类主导大类）"。这个表述可能被误读为"大类规律对品类有预测效果"。更精确的表述应该是"**大类规律对该品类基本等价**（因为该品类占据大类 85%）"——这是体型主导，不是预测力传导。

   **建议**：将"裁决"列改为：
   - 交流避雷器：**等价**（品类主导大类，非大类可预测品类）
   - 电容式电压互感器：**部分代表**（全月相关但非零月降至 0.50）
   - 交流支柱绝缘子：**不能代表**（非零月仅 0.37）

### ❌ 必须修复的错误

5. **[与 Phase 0/1 文档的逻辑脱节]** 文档的"建议"节（第六节）给出三个建议：
   - 放弃大类->品类层次化预测
   - 如果必须做，采用多任务学习
   - 探索项目级聚合

   但这些建议没有提及 Phase 1 数据稀疏度分析的核心推荐（**两阶段模型 + 批次事件特征**），也没有提及 Phase 0 项目逻辑步骤拆解中 VMD 架构已被判定为循环论证。读者如果按顺序阅读本文档（而不先看 Phase 1 的话），将得到一个独立但**不完整**的建议集。

   **建议**：在"建议"节开头增加一段前置声明：
   > "**前置约束**：本分析仅评估大类到品类的相关性是否能支持层次化预测。建模架构层面的审查（含 VMD 循环论证、两阶段模型设计、特征工程数据泄露等）参见项目逻辑步骤拆解.md（Phase 0）和数据稀疏度与合并方案.md（Phase 1）。本文档的建议基于'品类级独立建模'的前提。"

这是 3 篇文档中**信息密度最高、覆盖面最广**的一篇。评估了 18 个代码模块，涵盖了从数据采集到模型训练的全链路。评估标准清晰（贡献度/可复用性/技术合理性/论文出现位置），判定结论有具体的代码引用。但需要关注与影响因子增强方案的联动一致性。

2. **保留/需修改/需重写/废弃的四级判定清晰**：边界标准在第 17-21 行明确定义，"有根本缺陷且不可修"vs"需大修或重写"vs"需小修改"三者区分合理。

## 三、审查：影响因子增强方案.md

### 总体评价

这篇文档的方法论框架完整（诊断 -> 文献 -> 可行性矩阵 -> 优先级 -> 实施路线图）。但存在两个关键问题：(1) 与代码资产价值评估中"废弃"判定的一致性需要验证；(2) P0 因子的实现路径不够具体。

### ✅ 通过的审查点

1. **7 因子 Spearman 诊断有数据支撑**：表 1 的每个相关系数都有 p 值标注（***/**/*），数据范围明确（81 个月，非零月数标注）。

2. **"只有 project_count 是唯一有效因子"的诊断尖锐且正确**：rho=0.86-0.90 但这是同源泄漏特征（同一张 ECP 表聚合出来的），其余 6 个因子基本无效。这个诊断不能被轻描淡写——它直接影响论文的差异化定位（"首次引入外部因子"如果是首次引入无效因子，发表价值存疑）。

3. **文献整理全面**：CNKI/万方/IEEE/专利多源文献，划分为内部因素、外部因素、时序特征工程三类，每类都有具体的文献引用。

4. **候选因子可行性矩阵（第 3 节）覆盖了五大维度**：数据可获得性/采集成本/预期预测力/论文创新价值/综合建议。这个框架对下一步的决策有实际指导价值。

5. **实施路线图（第 6 节）有时间窗口、内容和交付物**，具备可操作性。

### 🔶 需要改进的问题

1. **[P0 因子实现路径不够具体 —— 关键遗漏]** 这是审核的重点。方案给出了因子名称和工作量估算，但没有给出**具体的实现路径和代码/数据库位置**：

   - **F01 (demand_lag12)**：方案说"从 material_demand_total 表生成"。实际上，这个特征**已经存在于 vmd-catboost/main.py 的特征工程中**（line 458: `lag12 = np.roll(demand_raw, 12)`），但存在 P-S8-2 数据泄露问题（在全部 81 个月上 roll 后切分）。方案没有指出：(a) lag12 已存在但数据泄露；(b) 修复泄露后 lag12 在测试期的值只能来自训练集窗口；(c) 这意味着 lag12 作为外部因子需要在 `preprocess_data` 重构后再测试效力。

   - **F04 (batch_count_by_month)**：方案说"从 bid_notices 按月+年聚合 COUNT"。但 `bid_notices` 表中已有 `notice_publish_time`（YYYY-MM-DD 格式）和 `bid_batch`、`bid_year` 字段——可以直接按月聚合。方案没有给出具体 SQL：
     ```sql
     SELECT substr(notice_publish_time, 1, 7) AS month,
            COUNT(*) AS batch_count
     FROM bid_notices WHERE category = '物资正刊'
     GROUP BY month ORDER BY month
     ```
     一眼就能验证可行性。

   - **F05 (batch_gini)**：当月 NOTICE 数量的季度 Gini 系数。这个描述有问题——Gini 系数衡量的是一个分布的不均匀程度，需要多个数据点才能计算。"当月"只有一个数据点（当月的 NOTICE 数量），无法计算 Gini。应该澄清是"滚动 N 个月窗口内的 Gini"还是"基于过去 12 个月的 Gini"。

   - **F06 (days_to_spring_festival)**：方案只说"chinese_calendar 库"，但没有说明：(a) 春节日期需要每年查询（非固定日期）；(b) 距春节天数可以是负数（春节后），需要明确正负的含义编码。

   总体而言，P0 因子（工作量标注为 0.3-1 人日）的"实现路径"一栏过于简略。每个因子如果只是给一个工具库名，读者无法判断实现难度是否真的只有 0.3 人日。

   **建议**：每个 P0 因子的"实现方式"列增加一个 `具体步骤/代码片段` 段，至少说明：(a) 从哪个表/API 获取数据；(b) 一个 SQL/Python 示例；(c) 需要注意的问题（如 lag12 的数据泄露已修复）。

2. **[与代码资产价值评估的"废弃"结论的矛盾 —— 重点审查项]**

   影响因子增强方案的附录（第 153-163 行）列出了"当前已知 R^2 对比"表，包括 VMD-CatBoost（R^2=0.07/0.43/0.48）、VMD-Transformer-CatBoost（R^2=0.34/0.52/0.44）、VMD-SVR（R^2=0.98/0.99/0.99）。表后有一个脚注：

   > "VMD-SVR 的极高 R2 可能存在于归一化域(非原始量纲)或存在数据泄露风险，需在最终验证中额外检查。"

   但是，代码资产价值评估已经明确判定 VMD-SVR（#15）为"废弃"——不仅是因为数据泄露，而且是**三因素叠加**（循环论证+RBF核+GridSearch）导致虚假高 R^2。影响因子增强方案中的描述"需在最终验证中额外检查"严重弱化了这个问题的严重性——这不是"需要检查"，而是**已经确认不可用**。

   同样，VMD-CatBoost（#12）和 VMD-Transformer-CatBoost（#13）在代码评估中也已判定为"废弃（循环论证）"，但影响因子增强方案没有提及这一点，读起来像是在暗示"修一修 R^2 还能用"。

   **建议（P0 修复）**：在影响因子增强方案附录的 R^2 表后，将当前脚注替换为：
   > "**方法论审查结论**（详见代码资产价值评估.md）：VMD-CatBoost、VMD-Transformer-CatBoost 和 VMD-SVR 三个模型已被判定为存在循环论证缺陷（IMF 作为特征预测 y，而 ΣIMF ≈ y），不应出现在论文中。VMD-SVR 的 R^2=0.98-0.99 是核方法+循环论证的产物，不具备预测意义。CatBoost baseline 是当前唯一方法学正确的模型。本文档的因子增强方案提升的是 CatBoost baseline 的 R^2，而非 VMD 系列模型的 R^2。"

3. **[预期 R^2 提升的数值过于精确]** 第 5 节表格给出了非常具体的 R^2 区间（如 P0 因子将交流避雷器 R^2 从 0.15 提升到 0.25-0.35）。但 10pp 的提升是基于什么？是文献基准的 0.07 R^2/因子（从附录中每个因子平均贡献 0.04 来估算），还是来自其他研究的类比？方案没有说明估算方法。

   **建议**：将"预期 R^2 提升估计"改为按因子类别的"保守/乐观"区间，而不是点估计：
   > "P0 因子预期 R^2 增量：自回归类因子（lag12/demand_roll3_mean/demand_yoy）基于时序预测文献的通用基准，保守估计新增 5-10pp；批次事件特征（batch_count/gini/spring_festival）缺乏直接类比文献，按保守/乐观分别估 3pp/10pp。"

4. **[遗漏：未讨论零值月对因子计算的影响]** P0 因子中 F01-F03 都是基于历史需求量的自回归特征。但三种物资有 29-31 个零值月（36-38%）。lag12 的"去年同期"如果是零值月，这个 lag 的值就是 0——但并不意味着"需求量真的为 0"，而可能意味着"去年同月没有发布公告"。这个歧义在方案中没有讨论。

   **建议**：在 P0 因子实现部分增加一条注意事项：
   > "自回归类因子（F01-F03）均基于 material_demand_total 的月度聚合数据，零值月的业务语义（无公告月 vs 真实零需求）尚未区分。在零值月语义未明确的情况下，lag12 的值可能受到系统性偏差影响。建议在实施阶段优先修复 demand_month 语义（P-S3-3）和零值区分（P-S4-2）后再评估这些因子的真实效力。"

5. **[遗漏：project_count 在 P0 方案中的角色不明确]** 影响因子增强方案的诊断结论明确指出 project_count 是"同源泄漏特征"。但 P0 因子方案中完全没有提到如何处理或替换 project_count。如果 project_count 仍是 top-1 最重要的 Spearman 因子，那么 P0 新增的因子能否替代它的预测力？

   **建议**：在诊断结论后增加一个判断："project_count 作为同源泄漏特征应保留在特征工程中（作为一个重要的采购规模代理），但论文的方法论部分应明确标注其为'内部衍生特征'而非'独立外部影响因子'。P0 新增因子的目标不是替换 project_count 的预测力，而是在其基础上增加独立的外部信息。"

### ❌ 必须修复的错误

6. **[与代码资产价值评估的"废弃"结论矛盾 —— P0 必须修复]** 详见上文改进问题 2。影响因子增强方案附录中 R^2 表对 VMD 系列模型的描述与代码评估的"废弃"判定严重不一致。如果读者先看影响因子增强方案再看代码评估，会产生困惑——到底是"需要验证"还是"已经废弃"？
