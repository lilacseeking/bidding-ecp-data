# ECP数据作为电力物资月度需求代理变量的验证分析

> 日期：2026-07-05
> 目标：验证ECP平台招标公告中的"数量"字段是否可以作为电力物资月度需求的代理变量

---

## 1. "数量"字段的业务含义 [高置信]

### 1.1 代码层面的发现

在 `src/pipeline.py` 的 `detect_column_map()` 函数（第60-122行）中，Excel列名检测逻辑如下：

- 列名 `"数量"` 被映射为 `quantity` 字段（第99行）
- 此 `quantity` 被赋值给 `bid_items.demand_quantity`（第620行）
- **关键证据**：同一Excel模板的货物清单中，还包含 `"首批交货日期"` 和 `"最后一批交货日期"` 两列（第100-103行），对应 `delivery_first` 和 `delivery_last` 字段

### 1.2 delivery_first 和 delivery_last 的解析与存储

这两个字段**被解析但不作为独立字段存储**。从 `parse_goods_list()` 函数（第568-636行）的逻辑可见：

```python
# 第601-611行：合并附加信息到remark
remark_parts = []
for key, label in [('voltage_level', '电压等级'),
                   ('delivery_method', '交货方式'),
                   ('delivery_first', '首批交货'),    # 解析了
                   ('delivery_last', '最后交货'),     # 解析了
                   ('project_name', '项目'),
                   ('tech_spec_code', '技术规范')]:
    v = safe_cell(row, col_map.get(key))
    if v:
        remark_parts.append(f"{label}:{v}")
```

**结论**：`delivery_first` 和 `delivery_last` 被拼接入 `remark` 字段（形式如 `"首批交货:2029-07-01; 最后交货:2029-07-31"`），不作为独立数据库列存在。

### 1.3 数据库实测验证

从 `ecp_data.db` 查询发现：
- 365,307 / 366,101 条记录（99.8%）的 `remark` 字段包含交货日期信息
- 交货日期跨度极大：从 0 天（同一天交货）到 793 天（超 2 年）

| 交货跨度 | 占比 |
|----------|------|
| 0-30天 | 71.9% |
| 30-90天 | 10.5% |
| 90-180天 | 5.5% |
| 180-365天 | 1.5% |
| 1-2年 | 10.4% |
| 2-3年 | 0.1% |

平均交货跨度约 59 天（约 1.9 个月）。

### 1.4 业务含义判定

**"数量"字段是"该批次的采购总量"，而非"月度需求量"。**

判定依据：
1. 一个招标公告对应的是某个批次的**整个采购包**，公告本身不包含月度拆分逻辑
2. 同一物资在同一公告中可能存在多行（对应不同项目/不同交货要求），每行的"数量"是该项目的该批次总需求
3. 货物清单同时包含"首批交货日期"和"最后一批交货日期"——如果数量已经是月度值，就不需要"首批/最后"来定义交货时间窗口
4. 从实际数据看，同一物资同一公告内不同行有不同的交货日期（如2020年某OPGW光缆公告，不同项目同一物资的交货日期分布在4月-6月的不同天数），说明每行数量对应的是项目层面的批次总量，按具体项目分批次交货

> **判定：确定** -- "数量"列是批次采购总量，不是月度需求。

## 2. demand_month 的赋值方式 [确定]

### 2.1 赋值逻辑

在 `phase4_parse()` 函数第453行：

```python
task['pub_time'].replace('-', '')[:6] if task.get('pub_time') else None,
```

**`demand_month` 是公告发布时间的前6位（YYYYMM格式），即公历月份。**

例如：公告 `notice_publish_time = "2026-07-03"`，则 `demand_month = "202607"`。

### 2.2 这意味着什么

当前数据处理将**公告发布时间**作为"需求月份"。但这个赋值存在逻辑问题：
- 招标公告的发布时间是招标流程的起点，而非物资实际需求时间
- 实际交货可能发生在公告发布后数月至数年（如前述，平均交货跨度1.9个月，最长可达26个月）
- "最后交货日期"可能落在公告发布后 2-3 年

**因此，将 `demand_month` 直接等同于"月度需求时间"是错误的。**

## 3. 公告发布节奏与月度需求的关系

### 3.1 实证分析：月度公告分布

从 `ecp_data.db` 查询 2019-11 至 2026-07 的物资正刊公告分布（311条）：

| 年份 | 公告数 | 月均 | 特点 |
|------|--------|------|------|
| 2020 | 46 | 3.8 | 数据采集开始年，覆盖不完整 |
| 2021 | 42 | 3.5 | 部分月份为0 |
| 2022 | 35 | 2.9 | 较少 |
| 2023 | 44 | 3.7 | 部分月份缺失 |
| 2024 | 51 | 4.3 | |
| 2025 | 50 | 4.2 | |
| 2026 | 32（至6月） | 5.3 | |

各年份几乎每月都有公告发布，但各月分布不均匀（范围1-12条/月）。这与国网的批次采购节奏一致 -- 输变电设备约6批/年，各批次交错分布在不同月份。

### 3.2 交货日期 vs 公告日期的错位

从 `remark` 字段提取的交货日期分析：
- 首批交货日期比公告发布时间晚 0-60 个月的都有
- 最后一批交货可能比公告发布晚 2-3 年（如：公告 2026-06-12 发布，最后交货 2028-12-31）
- 大型设备（抽水蓄能机组等）的首批交货甚至在公告后 5 年（如 2031-06-30）

### 3.3 同公告内多行数据的交货多样性

同一物资在同一公告中可有多个交货批次。例如：
```
公告 2020012023221987, 物资=OPGW光缆,48芯,G.652,90/112/45,铝包钢
  行1: 首批=2020-05-30, 最后=2020-05-30
  行2: 首批=2020-04-20, 最后=2020-05-20
  行3: 首批=2020-06-30, 最后=2020-06-30
  行4: 首批=2020-06-30, 最后=2020-06-30
  行5: 首批=2020-04-20, 最后=2020-06-15
```

这说明同一公告中的同一物资按**不同项目单位/不同交货时间**拆分为多行，每行的交货时间不同。

## 5. 关键发现：ECP数据 vs ERP数据

### 5.1 国家电网物资数据的三个层次

```
层次1: ECP公告中的"数量" → 批次采购总量（招标层面）
层次2: ERP中的"采购订单" → 中标后分解为订单（履约层面）  
层次3: ERP中的"领用/消耗" → 实际使用（消耗层面）
```

ECP公告数据 = 层次1 = "想要买多少"（计划值）
月度需求预测应该基于 = 层次3 = "实际用了多少"（实际值）

### 5.2 为什么要区分这两个概念

1. **时间错位**：ECP公告的"数量"是在某个时间点确定的一批采购总需求，而"月度需求"是每个月实际消耗的物资量
2. **数量含义不同**：一批协议库存的招标量覆盖的是未来 1-2 年的预估需求，而月度需求是月粒度的实际消耗
3. **预测目标的偏差**：用批次总量预测月度需求 = 用季度/年度计划预测月度执行，存在结构性偏差

## 6. 明确结论

### 结论：ECP数据中的"数量"不可以直接作为月度需求的代理变量

**原因**：

1. **"数量"是批次总量，不是月度需求** -- 这是根本性的数据结构问题，而非精度问题
2. **时间错配** -- 公告发布时间不等于需求时间，交货日期可能滞后数月至数年
3. **不同采购模式混叠** -- 同一月份可能混合了输变电批次招标（百公里电缆）、框架协议（预估1年量）、零星采购（少量多次）等不同模式
4. **文献共识** -- 已发表的物资预测论文主要使用ERP履约/领用数据，而非ECP招标公告数据

### 如果强行使用ECP数据作为月度代理变量的影响

- 月份间方差被过度放大（批次集中在几个月发布，导致这些月份"需求"异常高,其他月份甚至为0）
- 趋势失真：预测模型将学习到的是"批次公告节奏"而非"实际消耗趋势"
- 数量级偏差：公告中的数量涵盖的是整个交货期（可能数年）的需求，分配到单月会严重高估

## 7. 转换方案建议

### 7.1 基于交货日期的均匀分配法

利用已解析的 remark 字段中的"首批交货"和"最后交货"日期：

```
月度需求 = Σ(批次总量 / 交货跨度月数) × I(该月在交货区间内)
```

具体步骤：
1. 从 remark 提取每行的 `delivery_first` 和 `delivery_last`
2. 计算交货跨度 = `(delivery_last - delivery_first)` 的月数（最小 1 个月）
3. 将每行的数量均匀分配到交货跨度内的各月
4. 将所有行在同一个月的分配值求和

**优点**：合理利用了已有的交货时间窗信息
**局限**：均匀分配假设可能不成立（工程物资交货往往是前少后多或分批集中）

### 7.2 基于公告类型加权分配法

根据国网采购规律，不同类型的公告对应不同的需求覆盖期：

| 公告类型 | 覆盖期 | 分配策略 |
|----------|--------|----------|
| 批次招标(第N次) | 2-3个月 | 集中分配到未来 2-3 个月 |
| 协议库存框架 | 12个月 | 均分到未来 12 个月 |
| 零星采购 | 1-2个月 | 分配到下月 |

需要从公告标题中识别采购类型（当前已有 `bid_batch` 字段如"第一次"）。

### 7.3 层次化预测（推荐）

不直接转换ECP数据，而是将ECP数据作为**预测模型的"引导信号"**：

1. 使用ERP的历史月度履约/领用数据作为训练目标
2. 将ECP招标公告数据作为**外部特征**（如："过去N个月某物资的招标总量"作为影响因子）
3. 使用 `delivery_first` 作为需求释放的预期时间信号

### 7.4 论文中的实战方案

如果论文必须使用ECP数据（例如无法获取ERP数据），建议：

1. **明确声明数据局限**：在论文中说明数据来源为ECP批次招标数据，"月度需求"的定义是"该月公告涉及的批次总量"，而非实际消耗
2. **选用高频物资**：只选取出现频率 >= 12个月（全年覆盖）的物资，这类物资基本每月都有批次涉及，更接近真实月度需求模式
3. **做分布一致性验证**：如果可能，拿一小段ERP月度数据与ECP月度聚合数据做相关性分析，量化偏差

## 8. 后续建议

1. **数据分层标注**：在 `bid_items` 表增加 `delivery_first_month` 和 `delivery_last_month` 字段，从 remark 中解析出来
